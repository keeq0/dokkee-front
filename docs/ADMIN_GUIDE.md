# Инструкция администратора Dokkee

Инструкция для технического администратора, который сопровождает развёрнутую систему Dokkee. Описывает управление инфраструктурой, учётными записями пользователей, секретами, мониторинг сервисов и порядок действий при инцидентах. Подразумевается, что фронтенд и бэкенд развёрнуты по инструкциям `DEPLOYMENT.md` (фронтенд) и README бэкенда.

## 1. Состав системы и точки администрирования

Система состоит из независимо развёртываемых компонентов:

| Компонент          | Технология              | Что хранит / что делает                                       |
|--------------------|-------------------------|---------------------------------------------------------------|
| Фронтенд           | Vue 3, Nginx            | Статический бандл, отдаётся Nginx, проксирует `/api`, `/auth`.|
| Бэкенд             | Go (Gin)                | REST API: аутентификация, документы, результаты, профиль.     |
| База данных        | PostgreSQL 18           | `auth_credentials`, `user_profiles`, `user_balances`, `documents`, `analysis_results`, `audit_log`. |
| Объектное хранилище| MinIO (S3-совместимое)  | Исходные файлы документов в bucket из `S3_BUCKET`.            |
| AI-провайдер       | Внешний HTTP API        | Анализ обезличенного текста. URL и ключ — на стороне бэкенда. |

Все компоненты, кроме фронтенда, поднимаются `docker compose` из репозитория бэкенда. Фронтенд — отдельной командой `docker compose up -d frontend` или раздачей `dist/` через Nginx.

## 2. Учётные записи пользователей

Пользователи регистрируются самостоятельно через UI (`POST /auth/sign-up`). Учётные данные сохраняются в таблицах:

- `auth_credentials` — логин, хеш пароля (с солью).
- `user_profiles` — ФИО, email, телефон.
- `user_balances` — баланс на счёте.

### Просмотр и поиск пользователей

```sql
SELECT a.id, a.username, p.first_name, p.last_name, p.email, b.balance
FROM auth_credentials a
LEFT JOIN user_profiles p ON p.user_id = a.id
LEFT JOIN user_balances b ON b.user_id = a.id
ORDER BY a.id DESC
LIMIT 50;
```

### Блокировка пользователя

В текущей схеме отдельного флага блокировки нет. Чтобы временно прекратить доступ, изменяется хеш пароля на заведомо недостижимый:

```sql
UPDATE auth_credentials SET password_hash = 'blocked'
WHERE username = '<логин>';
```

После этого вход возвращает 401. Восстановление выполняется сбросом пароля.

### Сброс пароля

Самостоятельного сброса в UI нет. Администратор формирует новый хеш по той же схеме, что использует бэкенд (`bcrypt`/`sha` + соль из `salt`), и подставляет в `password_hash`. Пользователю передаётся новый пароль защищённым каналом.

### Принудительное завершение всех сессий

JWT-токены подписываются ключом `sign_key`. Изменение `sign_key` в `.env` бэкенда инвалидирует все ранее выданные токены — пользователи будут принудительно перенаправлены на форму входа.

```bash
cd ../dokkee-backend
sed -i 's|^sign_key=.*|sign_key=<новое значение>|' .env
make backend-down && make backend-up
```

## 3. Управление секретами

Секреты системы:

| Секрет           | Где задаётся                       | Назначение                                  |
|------------------|------------------------------------|---------------------------------------------|
| `salt`           | `.env` бэкенда                     | Соль для хеширования паролей.               |
| `sign_key`       | `.env` бэкенда                     | Подпись JWT-токенов.                        |
| `POSTGRES_PASSWORD` | `.env` бэкенда                  | Пароль БД.                                  |
| `S3_ACCESS_KEY`  | `.env` бэкенда                     | Ключ доступа к MinIO.                       |
| `S3_SECRET_KEY`  | `.env` бэкенда                     | Секрет доступа к MinIO.                     |
| `AI_API_KEY`     | `.env` бэкенда                     | Ключ внешнего AI-провайдера.                |
| Сертификаты TLS  | `/etc/letsencrypt/live/<domain>/`  | HTTPS на Nginx фронтенда.                   |

Ключ AI-провайдера хранится только на бэкенде — в бандл фронтенда не попадает.

### Ротация JWT-ключа

Регламент — раз в 90 дней или при инциденте. Алгоритм:

1. Сгенерировать новое значение длиной не менее 32 символов:
   ```bash
   openssl rand -hex 32
   ```
2. Заменить `sign_key` в `.env` бэкенда.
3. Перезапустить бэкенд (`make backend-down && make backend-up`).
4. Все активные токены становятся невалидными — пользователи входят повторно.

### Ротация AI-ключа

1. Получить новый ключ в кабинете AI-провайдера.
2. Обновить `AI_API_KEY` в `.env` бэкенда.
3. Перезапустить бэкенд: `make backend-down && make backend-up`.
4. Отозвать предыдущий ключ в кабинете провайдера.
5. Записать факт ротации в журнал секретов (дата, метка ключа; **само значение в журнал не пишется**).

### Ротация TLS-сертификатов

Сертификаты Let's Encrypt обновляются автоматически через `certbot`:

```bash
sudo certbot renew --dry-run     # проверка
sudo certbot renew                # реальное обновление
sudo systemctl reload nginx
```

## 4. Мониторинг сервисов

### Базовые команды

```bash
# Состояние контейнеров фронтенда
docker compose ps

# Состояние контейнеров бэкенда
docker compose -f ../dokkee-backend/docker-compose.yaml ps

# Логи фронтенда
docker compose logs -f frontend

# Логи бэкенда
docker compose -f ../dokkee-backend/docker-compose.yaml logs -f dokkee-backend

# Потребление ресурсов
docker stats
```

### Healthcheck

Healthcheck настроены у `frontend` (фронтенд), `dokkee-postgres` и `dokkee-minio` (бэкенд):

```yaml
healthcheck:
    test: ["CMD", "wget", "-qO-", "http://127.0.0.1:8080"]
    interval: 5s
    timeout: 3s
    retries: 30
    start_period: 30s
```

Статусы:

- `starting` — в окне `start_period`, контейнер ещё инициализируется.
- `healthy` — последняя проверка успешна.
- `unhealthy` — N подряд провалов. `depends_on: condition: service_healthy` блокирует запуск зависимых сервисов.

### Внешний мониторинг доступности

Для production-домена настроен внешний uptime-monitor (Uptime Kuma) с проверкой:

- `GET https://<domain>/` каждую минуту, ожидаемый код 200.
- `POST https://<domain>/auth/sign-in` каждые 5 минут с тестовой учётной записью, ожидаемый код 401 (без тела) либо 200 (с правильными учётными данными).
- TLS-сертификат: уведомление за 14 дней до истечения.

## 5. Резервное копирование

| Объект                  | Где лежит                       | Способ резервного копирования                      |
|-------------------------|----------------------------------|----------------------------------------------------|
| Исходный код фронтенда  | Git                              | GitHub + локальный mirror.                         |
| Исходный код бэкенда    | Git                              | GitHub + локальный mirror.                         |
| Production-бандл фронтенда | `/var/www/dokkee/releases/`   | Хранятся последние 10 релизов с timestamp.         |
| База данных             | Volume `pg_data`                 | Ежедневный `pg_dump`, хранение 30 дней.            |
| Файлы документов        | MinIO bucket из `S3_BUCKET`      | Ежедневный `mc mirror` в резервный bucket.         |
| Nginx-конфигурация      | `/etc/nginx/sites-available/`    | Git config-репозиторий.                            |
| TLS-сертификаты         | `/etc/letsencrypt/`              | Системный бэкап (`restic`/`rsnapshot`).            |
| `.env` бэкенда          | Файл на сервере                  | Зашифрованный бэкап в защищённое хранилище.        |

### Резервная копия БД

```bash
docker exec dokkee-postgres pg_dump -U $POSTGRES_USER $POSTGRES_DB | \
    gzip > /var/backups/dokkee/db-$(date +%F).sql.gz
```

### Резервная копия MinIO

```bash
mc mirror local/$S3_BUCKET backup/$S3_BUCKET
```

### Восстановление БД

```bash
gunzip -c /var/backups/dokkee/db-2026-05-14.sql.gz | \
    docker exec -i dokkee-postgres psql -U $POSTGRES_USER $POSTGRES_DB
```

### Откат бандла фронтенда

```bash
cd /var/www/dokkee
ls -la releases/                   # выбрать предыдущий релиз
ln -snf releases/2026-05-10-14-32 dist
sudo systemctl reload nginx
```

Время отката — несколько секунд (переключение симлинка).

## 6. Аудит и журналы событий

Бэкенд ведёт журнал действий пользователей в таблице `audit_log`. События:

- `USER_REGISTERED`
- `USER_LOGIN`
- `DOCUMENT_UPLOADED`
- `DOCUMENT_ANALYSIS_STARTED`
- `DOCUMENT_ANALYSIS_COMPLETED`
- `DOCUMENT_ANALYSIS_FAILED`
- `RESULT_ACCESSED`
- `PROFILE_ACCESSED`
- `PROFILE_UPDATED`
- `API_REQUEST`

Идентификаторы пользователя, документа и IP-адреса хранятся в виде хешей (`user_id_hash`, `doc_id_hash`, `ip_hash`). Это позволяет вести аудит без хранения персональных данных в журнале.

### Анализ активности пользователя

```sql
SELECT event_type, success, error_code, created_at
FROM audit_log
WHERE user_id_hash = encode(digest('<user_id>', 'sha256'), 'hex')
ORDER BY created_at DESC
LIMIT 100;
```

### Поиск аномалий

```sql
SELECT user_id_hash, COUNT(*) AS failures
FROM audit_log
WHERE success = false
  AND created_at > NOW() - INTERVAL '1 hour'
GROUP BY user_id_hash
HAVING COUNT(*) > 10
ORDER BY failures DESC;
```

Высокий процент ошибок у одного пользователя — повод проверить логи и при необходимости заблокировать учётную запись.

## 7. Реагирование на инциденты

### Уровни инцидентов

| Уровень | Признаки                                    | SLA реакции |
|---------|---------------------------------------------|-------------|
| P1      | Система недоступна, 5xx на главной странице | 15 минут    |
| P2      | Анализ документов не работает, ошибки AI    | 1 час       |
| P3      | Частичная деградация (медленные ответы)     | 4 часа      |
| P4      | Косметика, UX-баги                          | Следующий релиз |

### Сайт недоступен (P1)

1. Проверить доступность Nginx:
   ```bash
   curl -I https://<domain>/
   sudo systemctl status nginx
   sudo nginx -t
   ```
2. Проверить состояние контейнера фронтенда:
   ```bash
   docker compose ps frontend
   docker compose logs frontend | tail -100
   ```
3. Проверить TLS-сертификат:
   ```bash
   openssl s_client -connect <domain>:443 -servername <domain> < /dev/null | openssl x509 -dates -noout
   ```
4. После восстановления зафиксировать инцидент в журнале (issue tracker), приложить логи.

### API возвращает 5xx (P2)

1. Проверить состояние бэкенда:
   ```bash
   docker compose -f ../dokkee-backend/docker-compose.yaml ps dokkee-backend
   docker compose -f ../dokkee-backend/docker-compose.yaml logs dokkee-backend | tail -100
   ```
2. Проверить доступность БД и MinIO:
   ```bash
   docker exec dokkee-postgres pg_isready -U $POSTGRES_USER
   docker exec dokkee-minio mc ready local
   ```
3. При длительной недоступности БД — выполнить рестарт зависимостей.

### Анализ документов падает (P2)

1. Проверить статусы документов в очереди:
   ```sql
   SELECT id, status, error_msg, created_at
   FROM documents
   WHERE status = 'failed'
   ORDER BY created_at DESC LIMIT 20;
   ```
2. Если ошибки связаны с AI-провайдером — проверить квоту и валидность `AI_API_KEY`.
3. После восстановления повторно поставить документы в очередь:
   ```sql
   UPDATE documents SET status = 'queued', error_msg = NULL
   WHERE status = 'failed' AND created_at > NOW() - INTERVAL '1 hour';
   ```

### Утечка секрета

1. Незамедлительно отозвать скомпрометированный ключ у провайдера.
2. Сгенерировать новый ключ, обновить `.env` бэкенда.
3. Перезапустить сервис.
4. Изучить `audit_log` за период компрометации.
5. Уведомить ответственных и пользователей, если был доступ к их данным.

## 8. Регулярные регламентные работы

| Периодичность | Действие                                                                |
|---------------|-------------------------------------------------------------------------|
| Ежедневно     | Проверка `audit_log` на аномалии (массовые `success=false`).            |
| Ежедневно     | Резервная копия БД и MinIO.                                             |
| Еженедельно   | Проверка размеров `pg_data` и MinIO-bucket, очистка корзины при превышении лимита. |
| Еженедельно   | Просмотр логов на ошибки уровня `error` и выше.                         |
| Раз в 2 недели| `npm outdated` и `npm audit` для фронтенда, `go list -u -m all` для бэкенда. |
| Ежемесячно    | Проверка восстановления из резервной копии БД (на тестовом контуре).    |
| Раз в 90 дней | Ротация `sign_key` JWT и `AI_API_KEY`.                                  |
| Раз в 90 дней | Проверка прав доступа SSH к серверу, отзыв ключей уволенных сотрудников. |
| По SLA TLS    | Контроль автоматического продления `certbot renew` (за 14 дней до истечения). |
